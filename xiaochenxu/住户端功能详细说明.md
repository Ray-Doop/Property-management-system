# 住户端功能技术实现详细说明

## 后端技术栈与架构

### 核心技术栈
- **框架**：Spring Boot 3.x
- **安全框架**：Spring Security + JWT
- **持久层**：MyBatis + MySQL
- **缓存**：Redis
- **支付**：支付宝SDK
- **构建工具**：Maven
- **开发语言**：Java 17

### 架构设计
- **分层架构**：Controller → Service → Mapper → Database
- **前后端分离**：RESTful API设计
- **缓存策略**：多级缓存设计，支持缓存穿透、缓存击穿防护
- **事务管理**：声明式事务 + 编程式事务结合
- **多线程优化**：并行流处理缓存清理等耗时操作

### 核心设计模式
- **单例模式**：工具类和配置类
- **工厂模式**：对象创建和管理
- **代理模式**：Spring AOP实现日志和事务
- **策略模式**：不同支付方式处理

## 一、登录模块

### 1.1 验证码与账号登录
- **接口**：`/LoginRegister/captcha` - 获取验证码
  - **后端实现**：使用`CaptchaUtil`生成图形验证码，Base64编码返回，Redis缓存300秒（Key: `captcha:{uuid}`）
  - **参数校验**：UUID参数必填，验证码校验失败自动清理缓存

- **接口**：`/LoginRegister/userLogin` - 账号登录
  - **后端实现**：Spring Security AuthenticationManager认证，BCrypt密码匹配
  - **Token生成**：JWT Token含username、userId、role，有效期24小时
  - **Token管理**：Redis存储（Key: `login:user:{userId}`），支持单点登录
  - **日志记录**：记录登录IP、设备信息到`login_log`表

### 1.2 微信登录与注册
- **接口**：`/LoginRegister/wechatLogin` - 微信登录
  - **后端实现**：调用微信开放平台API验证code，获取openid
  - **用户处理**：openid不存在则自动注册，存在则直接登录
  - **Token生成**：与账号登录流程一致

- **接口**：`/LoginRegister/userRegister` - 用户注册
  - **后端实现**：密码BCrypt加密存储，初始status=0（待审核）
  - **数据校验**：手机号正则验证，用户名唯一性检查
  - **异步处理**：注册成功后异步发送欢迎通知

## 二、个人信息模块

### 2.1 信息管理
- **接口**：`/LoginRegister/GetUserData` - 获取用户信息
  - **后端实现**：`LoginRegisterService.getUserData()`，支持Redis缓存（20分钟）
  - **缓存策略**：用户信息变更时自动清理缓存
  - **权限控制**：只能获取当前登录用户信息

- **接口**：`/LoginRegister/UpdateUserData` - 修改用户信息
  - **后端实现**：`LoginRegisterMapper.updateUserData()`，事务管理
  - **缓存处理**：清除Redis用户信息缓存
  - **数据校验**：手机号、邮箱格式验证，敏感字段过滤

- **接口**：`/files/upload` - 文件上传
  - **后端实现**：`FileController.upload()`，支持分片上传
  - **存储方式**：本地文件系统存储，返回访问URL
  - **文件限制**：支持图片、文档等，单文件最大10MB
  - **异步处理**：异步生成缩略图（图片类型）

- **接口**：`/LoginRegister/UpdateAvatar` - 更新头像
  - **后端实现**：更新用户头像URL，清理相关缓存
  - **历史处理**：异步清理旧头像文件

- **接口**：`/LoginRegister/CheckOldPassword` - 验证旧密码
  - **前端调用**：密码修改页面，用户输入旧密码、新密码、确认新密码后，前端先验证新密码和确认密码是否一致
  - **请求体**：`{username, password: oldPassword}`
  - **后端实现**：
    1. `LoginRegisterService.checkOldPassword()`方法调用Spring Security的`AuthenticationManager.authenticate()`进行密码验证
    2. 验证失败返回错误信息，状态码401
    3. 验证通过返回成功信息，状态码200

- **接口**：`/LoginRegister/UpdatePassword` - 修改密码
  - **前端调用**：旧密码验证通过后调用，请求体为`{username, password: newPassword}`
  - **后端实现**：
    1. `LoginRegisterService.UpdatePassword()`方法接收用户信息
    2. 使用`BCryptPasswordEncoder.encode()`对新密码进行加密
    3. 调用`loginRegisterMapper.UpdatePassword(user)`执行SQL更新
    4. 清除相关缓存：
       - 删除`user:data:{userId}`缓存
       - 删除`user:username:{username}`缓存
       - 删除`userdetails:user:{username}`缓存
    5. 记录密码修改操作到审计日志
    6. 返回成功信息，状态码200
  - **前端处理**：
    1. 接收到修改成功响应后，清除本地存储的token和用户信息
    2. 提示用户"密码修改成功，请重新登录"
    3. 自动跳转到登录页面
  - **安全措施**：
    1. 新密码复杂度检查（长度≥8位，包含字母和数字）
    2. 防止SQL注入
    3. 操作日志记录
    4. 缓存清理确保数据一致性

## 三、公告模块

### 3.1 公告列表与详情
- **接口**：`/notice/List` - 公告列表
  - **后端实现**：`NoticeService.selectAllNotice()`，支持分页查询
  - **查询条件**：支持按标题、状态、发布时间筛选
  - **权限控制**：自动识别当前登录用户，返回对应权限的公告
  - **性能优化**：MySQL索引优化，大结果集分页处理

- **接口**：`/notice/{noticeId}` - 公告详情
  - **后端实现**：`NoticeService.selectById()`，关联查询附件信息
  - **阅读状态**：自动调用`/notice/read`标记已读
  - **计数更新**：公告阅读次数+1，异步更新统计数据

- **接口**：`/notice/read` - 标记已读
  - **后端实现**：`NoticeService.markAsRead()`，记录阅读时间到`notice_read`表
  - **幂等设计**：重复调用不重复记录
  - **异步处理**：异步更新公告阅读统计

- **接口**：`/notice/readStatus` - 获取阅读状态
  - **后端实现**：`NoticeService.readStatus()`，查询用户对指定公告的阅读状态
  - **缓存优化**：阅读状态缓存10分钟

### 3.2 公告附件管理
- **后端实现**：公告附件存储在`notice_attachment`表，关联noticeId
- **附件类型**：支持图片、文档、视频等多种类型
- **图片处理**：支持预览、缩放、旋转等操作
- **文件下载**：断点续传支持，大文件分片下载

### 3.3 公告发布与草稿
- **接口**：`/notice/publish` - 发布公告
  - **后端实现**：`NoticeService.publish()`，事务管理
  - **状态管理**：发布后status=1，自动设置发布时间
  - **附件处理**：批量保存附件信息，支持多附件上传

- **接口**：`/notice/saveDraft` - 保存草稿
  - **后端实现**：`NoticeService.saveDraft()`，status=0
  - **版本管理**：支持草稿多次保存，自动更新修改时间

### 3.4 公告统计
- **后端实现**：`NoticeService.summaryPage()`，统计公告发布数量、阅读情况
- **数据可视化**：支持按时间、类型统计，返回图表数据
- **实时更新**：异步更新统计数据，保证数据准确性

### 3.5 阅读公告情况（管理员功能）

#### 3.5.1 已读详情
- **前端**：管理员在公告列表页面，点击目标公告的"已读详情"按钮，调用`showReadDetail(row)`方法
  - **接口调用**：发送GET请求到`/notice/readTable`接口，参数为`{noticeId, readStatus: 1, keyword, page, size}`
  - **请求参数**：
    - noticeId：公告ID，必填
    - readStatus：固定为1（已读）
    - keyword：搜索关键词（用户名、手机号），可选
    - page：页码，默认1
    - size：每页条数，默认10

- **后端**：`NoticeController.readTable()`方法处理请求
  - **业务逻辑**：`NoticeService.readTable()`调用`NoticeMapper.getReadTable()`执行SQL查询
  - **数据来源**：关联查询`notice_read`表和`user`表，筛选条件为`read_status = 1`
  - **返回数据**：包含已读用户ID、用户名、手机号、阅读时间等信息的分页列表
  - **性能优化**：MySQL索引优化，大结果集分页处理

- **前端**：使用`el-dialog`组件显示已读详情弹窗，标题为"已读详情"，内部使用`el-table`组件展示已读列表
  - **表格列**：用户名、手机号、阅读时间
  - **样式设计**：已读状态显示为绿色"已读"
  - **分页功能**：底部显示分页控件，支持页码跳转和每页条数调整
  - **导出功能**：支持将已读详情导出为Excel文件

#### 3.5.2 未读详情
- **前端**：管理员在公告列表页面，点击目标公告的"未读详情"按钮，调用`showUnreadDetail(row)`方法
  - **接口调用**：发送GET请求到`/notice/readTable`接口，参数为`{noticeId, readStatus: 0, keyword, page, size}`
  - **请求参数**：
    - noticeId：公告ID，必填
    - readStatus：固定为0（未读）
    - keyword：搜索关键词（用户名、手机号），可选
    - page：页码，默认1
    - size：每页条数，默认10

- **后端**：`NoticeController.readTable()`方法处理请求
  - **业务逻辑**：`NoticeService.readTable()`调用`NoticeMapper.getReadTable()`执行SQL查询
  - **数据来源**：查询`user`表中未在`notice_read`表中记录的用户，筛选条件为`read_status = 0`
  - **返回数据**：包含未读用户ID、用户名、手机号等信息的分页列表
  - **性能优化**：MySQL索引优化，使用NOT IN或LEFT JOIN查询提高效率

- **前端**：使用`el-dialog`组件显示未读详情弹窗，标题为"未读详情"，内部使用`el-table`组件展示未读列表
  - **表格列**：用户名、手机号
  - **样式设计**：未读状态显示为红色"未读"
  - **分页功能**：底部显示分页控件，支持页码跳转和每页条数调整
  - **导出功能**：支持将未读详情导出为Excel文件

#### 3.5.3 阅读率统计
- **前端**：公告列表页面显示每个公告的阅读率，使用进度条可视化展示
  - **数据来源**：后端返回的公告列表中包含`readCount`（已读人数）和`totalCount`（总人数）
  - **计算方式**：阅读率 = (readCount / totalCount) * 100%
  - **样式设计**：
    - 阅读率达到100%显示绿色对勾
    - 阅读率低于100%显示蓝色进度条
    - 进度条长度根据阅读率动态调整

- **后端**：`NoticeService.selectAllNotice()`方法计算阅读率
  - **业务逻辑**：
    1. 查询公告基本信息
    2. 统计该公告的已读人数（`notice_read`表中记录数）
    3. 统计总用户数（`user`表中有效用户数）
    4. 计算阅读率，四舍五入保留一位小数
  - **性能优化**：使用子查询或联表查询一次性获取所有数据

## 四、出行码模块

### 4.1 出行码生成与刷新
- **接口**：`/travel-pass/issue` - 生成出行码
  - **后端实现**：`TravelPassServiceImpl.issue()`，生成唯一recordId
  - **二维码生成**：使用`QRCodeUtil`生成Base64二维码，含recordId和过期时间
  - **数据存储**：记录到`travel_pass_record`表，状态为"有效"
  - **过期策略**：默认24小时过期，可配置
  - **异步处理**：异步发送出行码生成通知

- **接口**：`/travel-pass/refresh` - 刷新出行码
  - **后端实现**：`TravelPassServiceImpl.refresh()`，更新时间戳
  - **幂等设计**：同一recordId短时间内多次刷新只更新一次
  - **状态检查**：已过期的出行码不允许刷新
  - **日志记录**：记录刷新操作到审计日志

- **接口**：`/travel-pass/myRecords` - 查询出行记录
  - **后端实现**：`TravelPassServiceImpl.getMyRecords()`，支持分页
  - **查询条件**：支持按时间范围、状态筛选
  - **权限控制**：只能查询当前登录用户的出行记录
  - **性能优化**：MySQL索引优化，大结果集分页处理

### 4.2 出行码验证
- **接口**：`/travel-pass/verify` - 验证出行码
  - **后端实现**：`TravelPassServiceImpl.verify()`，检查有效性和过期时间
  - **验证逻辑**：
    1. 检查recordId是否存在
    2. 检查状态是否为"有效"
    3. 检查是否在有效期内
    4. 记录验证日志
  - **响应设计**：返回验证结果和用户基本信息

### 4.3 出行码统计
- **后端实现**：`StatsService.getTravelPassStats()`，统计出行码使用情况
- **统计维度**：按日、周、月统计生成量、验证量
- **数据可视化**：返回图表数据，支持导出
- **实时更新**：异步更新统计数据

## 五、报修模块

### 5.1 报修提交与管理
- **接口**：`/repair/categories` - 获取报修类别
  - **后端实现**：`RepairService.selectCategories()`，Redis缓存60分钟
  - **缓存策略**：
    1. 先查Redis缓存
    2. 未命中则加锁查询数据库
    3. 缓存空值防止缓存穿透
    4. 多线程缓存清理
  - **数据来源**：`repair_category`表，支持多级分类

- **接口**：`/repair/submit` - 提交报修
  - **后端实现**：`RepairService.addRepairOrder()`，事务管理
  - **数据校验**：
    1. 类别ID必填且存在
    2. 标题、描述不能为空
    3. 图片数量限制（最多3张）
  - **附件处理**：调用`/files/upload`上传图片，返回URL列表
  - **状态管理**：初始状态为"待处理"
  - **日志记录**：记录提交时间、用户ID到`repair_order`表
  - **缓存清理**：清除用户维修单列表缓存

- **接口**：`/repair/myRepair` - 查询我的报修
  - **后端实现**：`RepairService.selectMyRepair()`，Redis缓存1分钟
  - **查询条件**：支持按状态、时间范围筛选
  - **分页处理**：PageHelper分页，默认每页10条
  - **权限控制**：只能查询当前登录用户的报修单

### 5.2 报修操作
- **接口**：`/repair/detail/{orderId}` - 获取报修详情
  - **后端实现**：`RepairService.getRepairDetail()`，Redis缓存15分钟
  - **关联查询**：
    1. 报修单基本信息
    2. 附件列表
    3. 处理记录
    4. 评价信息
  - **缓存策略**：
    1. 先查Redis缓存
    2. 未命中加锁查询数据库
    3. 缓存空对象防止缓存穿透

- **接口**：`/repair/evaluate` - 评价报修
  - **后端实现**：`RepairService.evaluateRepair()`，事务管理
  - **数据校验**：
    1. 评分必填（1-5分）
    2. 评价内容可选
    3. 只能评价已完成的报修单
  - **状态更新**：报修单状态更新为"已评价"
  - **缓存清理**：清除报修详情和列表缓存

- **接口**：`/repair/cancel/{orderId}` - 取消报修
  - **后端实现**：`RepairService.cancelRepair()`
  - **状态检查**：只能取消"待处理"状态的报修单
  - **日志记录**：记录取消原因和时间
  - **缓存清理**：清除相关缓存

### 5.3 维修人员管理
- **接口**：`/repair/findWorkers` - 查询维修人员
  - **后端实现**：`RepairService.findWorkers()`，Redis缓存30分钟
  - **查询条件**：按维修类别筛选
  - **数据来源**：`repair_worker`表，关联维修类别

- **接口**：`/repair/acceptByWorker` - 维修人员接单
  - **后端实现**：`RepairService.acceptByWorker()`，事务管理
  - **状态更新**：报修单状态更新为"处理中"
  - **记录生成**：生成`repair_assignment`记录
  - **缓存清理**：清除相关缓存

### 5.4 维修进度更新
- **接口**：`/repair/updateStatus` - 更新维修状态
  - **后端实现**：`RepairService.updateStatus()`

  - **后端实现**：`RepairService.replyEvaluation()`

  - **后端实现**：`PostService.addToCollection()`
  - **幂等设计**：重复收藏不重复记录
  - **数据存储**：记录到`post_collection`表
  - **计数更新**：帖子收藏数+1，异步更新

- **接口**：`/Forum/delCancelCollect` - 取消收藏
  - **后端实现**：`PostService.delCancelCollect()`
  - **幂等设计**：重复取消不报错
  - **计数更新**：帖子收藏数-1，异步更新
  - **缓存清理**：清除帖子详情缓存

### 6.3 帖子审核与管理
- **接口**：`/Forum/audit` - 帖子审核
  - **后端实现**：`PostService.audit()`，支持批量审核
  - **状态流转**：
    1. 待审核 → 已发布
    2. 待审核 → 已拒绝
  - **通知机制**：异步发送审核结果通知
  - **日志记录**：记录审核人、审核时间、审核理由

## 七、物业缴费模块

### 7.1 账单与支付
- **接口**：`/api/fee/billsById` - 查询账单列表
  - **后端实现**：`FeeService.getBillsById()`，支持分页
  - **查询条件**：支持按residenceId、账单类型、支付状态筛选
  - **数据来源**：`fee_bill`表，关联`residence`表
  - **权限控制**：只能查询当前登录用户关联的账单
  - **性能优化**：MySQL索引优化，大结果集分页处理

- **接口**：`/api/pay/create` - 创建支付订单
  - **后端实现**：`PayService.createOrder()`，事务管理
  - **数据校验**：
    1. 账单ID必须存在且未支付
    2. 支付金额必须与账单金额一致
  - **订单生成**：生成唯一订单号，记录到`pay_order`表
  - **支付表单**：构造支付宝支付表单，返回前端
  - **缓存处理**：记录支付订单到Redis，有效期30分钟

- **接口**：`/api/pay/notify/alipay` - 支付宝异步通知
  - **后端实现**：`PayController.alipayNotify()`，异步处理
  - **签名验证**：支付宝签名验证，防止伪造通知
  - **状态更新**：
    1. 更新`pay_order`表支付状态
    2. 更新`fee_bill`表支付状态
    3. 记录支付时间、交易号
  - **幂等设计**：重复通知只处理一次
  - **日志记录**：详细记录通知内容和处理结果
  - **通知机制**：异步发送支付成功通知

- **接口**：`/api/pay/result` - 查询支付结果
  - **后端实现**：`PayService.getPayResult()`
  - **查询逻辑**：
    1. 先查Redis缓存
    2. 未命中则查数据库
    3. 调用支付宝API验证最终状态
  - **响应设计**：返回支付状态、交易号、支付时间等

### 7.2 缴费统计与报表
- **接口**：`/api/fee/stats` - 缴费统计
  - **后端实现**：`StatsService.getFeeStats()`
  - **统计维度**：按时间、账单类型、支付方式统计
  - **数据可视化**：返回图表数据，支持导出Excel
  - **权限控制**：只能查看当前登录用户的统计数据

### 7.3 缴费提醒
- **后端实现**：`NoticeScheduler`定时任务，每天检查到期未支付账单
- **提醒方式**：站内信通知
- **触发条件**：账单到期前7天、3天、1天分别提醒
- **日志记录**：记录提醒发送时间、接收用户
